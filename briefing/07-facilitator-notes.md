# 진행자 노트

## 전체 톤

- **전도사가 아니라 실무자**로서 말하기
- "AI 좋습니다" 대신 "이렇게 쓰면 이런 결과가 나오더라"
- 한계와 실패 경험도 솔직하게 공유 → 신뢰도 상승

---

## 청중 유형별 대응

### 회의론자 ("그거 별로던데")
- 직접 써본 경험이 있으므로 경험을 존중
- "언제, 어떤 상황에서 쓰셨어요?" → 맥락에 따라 결과가 달라짐을 설명
- 데모에서 직접 요구사항을 던지게 하면 가장 효과적

### 열성 지지자 ("다 AI로 하면 되지 않나")
- 과도한 기대를 현실적으로 조절
- "80/20 모델" 강조 — AI는 도구지 마법이 아님
- 코드 리뷰와 검증의 중요성은 오히려 더 커짐

### 불안해하는 사람 ("우리 일자리는?")
- 직접 다루기보다는 자연스럽게 프레이밍
- "AI가 개발자를 대체하는 게 아니라, AI를 쓰는 개발자가 안 쓰는 개발자를 대체"
- **판단, 설계, 사용자 이해**는 여전히 사람의 영역임을 강조

### 관리자/리더
- 생산성 수치보다 **품질 향상** 강조가 더 지속가능
- "테스트 커버리지가 올라갑니다", "접근성 빠뜨리는 게 줄어듭니다"
- 도입 비용(라이선스비) vs 효과를 구체적 숫자로

## 예상 Q&A

### "어떤 AI 도구를 써야 하나?"

> 팀 상황에 따라 다르지만, 추천 조합:
> - **입문**: GitHub Copilot (가장 낮은 진입 장벽)
> - **중급**: Cursor (에디터 통합이 잘 됨)
> - **고급**: Claude Code (터미널 기반, 에이전트 모드 강력)
> - 하나를 정해서 팀 전체가 같은 도구를 쓰는 게 경험 공유에 유리

### "AI가 만든 코드에 버그가 있으면 누가 책임지나?"

> 코드 리뷰를 통과시킨 사람의 책임. 이건 사람이 짠 코드든 AI가 짠 코드든 동일.
> AI는 도구일 뿐이고, **커밋하는 사람이 그 코드의 오너**.

### "사수 없이 주니어가 AI만 쓰면 실력이 안 늘지 않나?"

> 좋은 질문. 두 가지 관점:
> 1. AI가 생성한 코드를 **읽고 이해하는 과정** 자체가 학습
> 2. 다만, 기초 없이 AI에만 의존하면 위험 — AI를 "정답"이 아닌 "참고 자료"로 활용하도록 가이드 필요
> 3. 오히려 AI에게 "왜 이렇게 했어?" 질문하면 좋은 설명을 받을 수 있음

### "기존 코드베이스가 레거시인데 AI가 이해할 수 있나?"

> L1은 어려움. L2/L3는 가능 — 코드를 직접 읽히고 설명을 요청하면 됨.
> 레거시 이해 → 마이그레이션 계획 → 점진적 변환에 AI가 특히 유용.

### "보안 심사를 통과해야 하는데 가능한가?"

> 엔터프라이즈 플랜은 대부분 SOC2, GDPR 준수.
> 보안팀에 전달할 자료: 각 도구의 데이터 처리 정책, 코드 학습 제외 조항.
> 필요하면 자체 호스팅 옵션도 있음 (비용은 높음).

## 데모 비상 대응

| 상황 | 대응 |
|------|------|
| 네트워크 끊김 | 미리 녹화한 영상으로 전환, "실제 화면과 동일합니다" |
| AI 응답이 이상함 | "이런 경우에는 프롬프트를 수정해서 다시 보내면 됩니다" → 수정 후 재시도 |
| AI 응답이 너무 느림 | "보통은 몇 초인데, 서버 상태에 따라 다릅니다" → 미리 준비한 결과물 사용 |
| 생성된 코드에 에러 | "에러를 AI에게 다시 보내면?" → 자체 디버깅 시연 → 오히려 좋은 데모 |
| 청중이 무반응 | "직접 해보시겠어요?" → 참여 유도. 실패해도 경험이 됨 |
