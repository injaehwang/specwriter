01. 왜 지금인가

AI는 코드 자동완성이 아니라, 설계→구현→검증 전 과정의 협업 파트너다.
FE는 반복 패턴이 많고, 프레임워크 변화가 빠르며, 빠른 시각적 피드백이 중요해서 AI 활용과 특히 잘 맞는다.

FE에서 특히 중요한 이유

02. AI 도입 3단계 성숙도 모델

AI 도입에는 단계가 있다. 대부분의 팀은 L1에 머물러 있지만, 실질적 효과는 L2부터 나타난다.

전체 그림

graph LR L1["L1 자동완성 (보조)"] L2["L2 대화형 협업 (협업)"] L3["L3 에이전트 위임 (주도)"] L1 -->|"맥락 제공"| L2 L2 -->|"스펙 기반"| L3

L1: 자동완성

코드를 타이핑하면 AI가 다음 줄을 제안. 현재 파일 수준의 맥락만 이해하며, 프로젝트 전체 구조는 모른다.

sequenceDiagram participant D as 개발자 participant AI as AI D->>AI: 코드 타이핑 중... AI-->>D: 다음 줄 제안 D->>AI: Tab (수락)

L2: 대화형 협업

개발자가 의도와 맥락을 설명하면 AI가 구조적 결과물을 생성. 개발자는 What, AI는 How.

sequenceDiagram participant D as 개발자 participant AI as AI D->>AI: 의도 + 맥락 + 요구사항 전달 AI-->>D: 구조적 결과물 생성 D->>AI: "여기 수정해줘" AI-->>D: 수정된 결과물

L3: 에이전트 위임

스펙을 주면 AI가 프로젝트를 직접 분석하고, 파일 생성/수정, 테스트 실행까지 수행. 사람은 최종 리뷰어.

sequenceDiagram participant D as 개발자 participant AI as AI participant FS as 프로젝트 D->>AI: 스펙/요구사항 전달 AI->>FS: 프로젝트 분석 AI->>FS: 파일 생성/수정 AI->>AI: 테스트 실행 + 디버깅 AI-->>D: 완성된 결과물 D-->>D: 리뷰 + 승인

05. 도입 시 흔한 실패 패턴

AI를 도입한 팀에서 자주 듣는 불만 5가지와, 왜 그런 일이 생기는지, 어떻게 해결하는지.

1. "AI가 만든 코드가 우리 프로젝트랑 안 맞아요"

AI는 우리 프로젝트의 컨벤션을 모른다. 프롬프트에 사용할 라이브러리, 기존 컴포넌트, 스타일을 명시하면 결과가 달라진다.

2. "결국 사람이 다 고쳐야 해요"

AI는 초안을 빠르게 만드는 도구이고, 사람은 판단하고 다듬는 역할이다.
80%를 30초에 받고 20%를 10분에 다듬기 vs 처음부터 100%를 40분에 만들기.

3. "같은 걸 시켜도 사람마다 결과가 달라요"

팀에 공유된 기준이 없으면 같은 요청에도 각자 다른 결과가 나온다. 프로젝트 컨벤션 파일을 AI에 연결하면 해결된다.

graph TD subgraph "컨벤션 없이" A1["개발자 A: 폼 만들어줘"] --> R1[VeeValidate] A2["개발자 B: 폼 만들어줘"] --> R2[FormKit] A3["개발자 C: 폼 만들어줘"] --> R3[직접 v-model] end subgraph "컨벤션 공유 시" B1["개발자 A: 폼 만들어줘"] --> R4[VeeValidate + zod] B2["개발자 B: 폼 만들어줘"] --> R4 B3["개발자 C: 폼 만들어줘"] --> R4 end

4. "회사 코드가 AI 학습에 쓰이는 거 아닌가요?"

유료/엔터프라이즈 플랜은 코드를 학습에 사용하지 않는다. 무료 플랜은 사용될 수 있으므로 업무용은 반드시 유료 플랜.

5. "한번 써봤는데 별로였어요"

최소 2주간 매일 써야 본인만의 활용 패턴이 만들어진다. 작은 것부터 시작하고, 팀 내 주간 공유로 서로의 활용법을 배운다.
